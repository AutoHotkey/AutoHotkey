; This file boosts performance by telling the linker how to physically order the functions inside the EXE.
; When functions that call each other (or that are called together) are near each other, they cooperate better
; with the CPU cache.  I believe this is because the cache and/or OS has a certain memory block size such as 4 KB.
; So when a block gets brought into the cache, it tends to bring in nearby functions with it.  Since there's only
; a small number of blocks available, having strongly related code sections adjacent to each other improves the
; percentage of "good code" in the cache.

; Putting small, frequently-called functions (even ones that don't call each other) adjacent to each other
; might be more beneficial than sandwiching big functions with small functions they call.

; Testing shows that semicolons can be used for comments, even at the end of a line.

; Info from http://www.kegel.com/mangle.html (not necessarily complete or accurate)
; Operator names are represented by a question mark followed by a single character rather than being spelled out:
; myclass::myclass          ?0
; myclass::~myclass         ?1
; myclass::operator new     ?2
; myclass::operator delete  ?3
; myclass::operator=        ?4
; myclass::operator+        ?H
; myclass::operator++       ?E

; Data Type Codes
; signed char    C
; char           D
; unsigned char  E
; short          F
; unsigned short G
; int            H
; unsigned int   I
; long           J
; unsigned long  K
; float          M
; double         N
; long double    O
; pointer        P (see below)
; array          Q (see below)
; struct/class   V (see below)
; void           X (terminates argument list)
; elipsis        Z (terminates argument list)

; NOTE: A CLEAN+BUILD is usually necessary to properly put changes made in this file into effect, at least in MSVC++ 7.1
